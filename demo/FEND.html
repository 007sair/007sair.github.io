MIA FEND

针对目前mia前端在工作中用到以及即将用到的技术，做一次整理，如下：

框架/库

前端工程第一个任务是根据项目、业务特征选择技术。 mia作为一家电商公司，前端技术并非从0开始，现就目前的情况做如下分析：

JS

库：pc端沿用jQuery（v1.8.3），移动端沿用zepto（v1.1.4）；

框架：在未来可以使用ReactJS作为专题系统模型使用；

CSS

库：css的库很多，学习成本低，可以用作学习或工作使用。如：Animate.css、Normalize.css等等

Sass作为团队目前正在使用的工具，在经历多个项目后，base库已相对成熟。但还需注意以下几点：

  • 编译：sass依赖于工具的编译，可以使用图形界面工具koala或构建工具进行编译。
  • 文件管理：倘若完全依赖于sass放弃css，需有一套完整的文件管理策略。良好的文件管理能提高团队的开发效率和代码的复用率。
  • 规范：没有规范就无法形成团队开发，这样不便于共同维护和共同提高。

自动化构建

前端的构建优化以前都是靠人为操作，自nodeJS出现后，自动化的工作方式产生并出现一系列的构建工具、包管理工具。

构建工具包括：Gulp、Grunt、百度FIS；

包管理工具：webpack、Broeserify；

Gulp

在经过多重选择后，还是选择了最为熟悉的gulp，它基本能处理常用功能：文件压缩、合并、图片压缩、雪碧图、iconfont构建、打包等

更深层次的依赖打包我觉得目前还用不上，可以等到js相对复杂时再做研究。

模块化

JS模块化方案很多，AMD/CommonJS/UMD/ES6 Module等，对应的框架和工具也很多，常见的为SeaJS和RequireJS

两种模块化的方式都存弊端，具体可参考： - 再谈 SeaJS 与 RequireJS 的差异

以目前的情况，可以先使用seaJS，毕竟国人开发，并且团队内已经有人使用过。

CSS模块化开发基本都是在less、sass、stylus等预处理器的特性支持下实现的。在完善了sass的使用和规范后，css的模块化也会相对简单。

PS：并非最新的技术就是最好的技术，需根据实际业务需要选择库和框架。
